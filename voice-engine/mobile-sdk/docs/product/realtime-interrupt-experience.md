# 实时语音打断体验整理

本文整理当前实时语音通话里的“打断”逻辑和体验问题。这里的打断不是结束会话，也不是静音，而是用户在 AI 播报过程中插话，系统应尽快停下当前播报，并把用户后续表达作为新的对话轮次处理。

## 1. 用户体验目标

用户感知上，打断应满足 4 点：

- 用户一开口，AI 当前声音尽快停下。
- 用户不用说固定口令，只要明显开始表达，就可以插话。
- 打断后系统进入“我停下了，你继续说”的状态，而不是沉默卡住。
- 旧播报不能在用户说完后重新漏出来。

按钮打断和语音打断的体验目标不同：

| 类型 | 用户意图 | 期望反馈 |
| --- | --- | --- |
| 语音打断 | 用户正在插话、纠正或补充 | 当前播报立刻降低存在感，继续收用户声音 |
| 按钮打断 | 用户明确要求停止当前播报 | 当前播报停止，界面提示“已停下，继续说” |
| 静音 | 用户暂时关闭输入和播报 | 停止采集麦克风，并停止播放 |
| 结束通话 | 用户结束整个 Voice Session | 关闭会话并生成报告 |

## 2. 当前实现链路

当前逻辑分为前端本地播放控制、前端自动打断判断、后端上游打断三层。

### 2.1 前端本地播放

`apps/web/src/app/App.tsx` 里维护几个关键状态：

| 状态 | 作用 |
| --- | --- |
| `playbackSourcesRef` | 当前浏览器已经排队或正在播放的 TTS 音频源 |
| `playbackMutedRef` | 静音时丢弃后续音频 |
| `activeAssistantOutputIdRef` | 当前 assistant 输出的 `outputId` |
| `discardedAssistantOutputIdsRef` | 已被打断、需要丢弃的旧 `outputId` |
| `assistantAudioActiveUntilRef` | TTS 音频后保留一段“助手仍在输出”的宽限时间 |
| `bargeInRef.recentSpeechCandidateAt` | 播放中检测到用户可能在说话的时间 |
| `forceNextAssistantTurnRef` | 打断后让下一条 assistant 文本作为新轮次展示 |

执行打断时，前端会：

1. 把当前活跃 `outputId` 加入 `discardedAssistantOutputIdsRef`。
2. 设置 `forceNextAssistantTurnRef=true`。
3. 停止本地播放队列。
4. 按需通过 WebSocket 发送 `{ "type": "interrupt", "targetOutputId": "..." }`。

收到后续 assistant 文本或音频时，如果它的 `outputId` 已在丢弃集合中，前端直接丢弃。文本和音频使用同一个判断边界。

### 2.2 前端自动打断判断

麦克风采集时，如果当前正在播放，前端会根据本地音量阈值判断是否“疑似用户说话”。这个阶段只记录候选时间，不直接停止本地播报，避免拍桌子、碰麦克风这类噪音误停。

真正通知上游打断发生在收到 ASR 文本后：

- 短口令命中，例如“停”“等一下”“打断”“stop”。
- 文本不是简单语气词。
- 文本不像助手播报内容的回声。
- 文本长度较长，或在本地检测到人声候选窗口内。

这意味着当前语音打断仍以 ASR 确认为准：本地音量候选只帮助判断短文本是否可能是插话，真正停播和上游 interrupt 在 ASR 文本确认后发生。

### 2.3 后端上游打断

`apps/api/realtime/gateway.py` 收到 `interrupt` 后会：

1. 读取或回退当前 `targetOutputId`。
2. 清空当前流式文本缓存。
3. 向火山实时语音发送 `CLIENT_INTERRUPT`。
4. 给前端返回 `interrupt_ack`，其中带 `targetOutputId`。

后端不再长期 suppress assistant 文本。收到新的 ASR 或文字输入后，下一条 assistant 输出必须进入新的 `outputId`，由前端按 `outputId` 判断旧输出是否丢弃。

## 3. 体验问题

### 3.1 用户开口后，AI 不一定马上停

当前本地音量检测只记录候选，不直接停止播放。用户插话后，仍可能听到 AI 继续说，直到 ASR 文本回来并触发自动打断。

这个取舍是为了避免拍桌子、碰麦克风、咳嗽等噪音直接停掉播报。后续如果接入真正 VAD，可以再把“高置信人声”升级为先本地停音。

### 3.2 旧版 suppress 容易压掉正确回复

旧版自动打断的实际顺序是：

1. 后端收到上游 ASR。
2. 后端把 ASR 发给前端。
3. 前端判断需要打断。
4. 前端再发 `interrupt` 给后端。
5. 后端设置 `suppress_assistant_until_user_input=true`。

问题是：触发打断的用户输入已经发生了。如果上游随后返回的 assistant 正是回答这次用户输入，后端仍可能把它当成“旧播报残留”压掉，导致用户说完后没有回应。

当前修正方向是取消长期 suppress。后端只保证用户输入后的下一条 assistant 使用新 `outputId`，旧输出是否丢弃由前端的 `discardedOutputIds` 统一判断。

### 3.3 打断按钮没有区分“正在播报”和“无播报”

当前按钮在通话中一直可点击。用户误点时，前端和后端都会进入打断状态，但实际上没有需要停止的当前播报。

体验风险：

- 界面显示已打断，但用户不知道下一步要不要继续说。
- 后端可能等待下一次用户输入才恢复 assistant 文本。
- 用户感知像“点了按钮以后卡住”。

### 3.4 文本和音频的丢弃边界不一致

旧版后端会 suppress assistant 文本和 TTS sentence，但仍会转发 `TTS_RESPONSE` 音频帧。前端依赖 `interruptingPlaybackRef` 来丢弃这些音频。

当前修正方向是文本和音频都带 `outputId`，前端只按 `discardedAssistantOutputIdsRef` 丢弃旧输出。

### 3.5 打断态承担了太多语义

旧版 `interruptingPlaybackRef` 同时表示：

- 当前已经停止本地播放。
- 后续音频需要丢弃。
- 正在等待用户新输入。
- assistant 输出是否仍活跃。

这些语义混在一个布尔值里，导致不同分支都在清理它。当前第一版已把核心丢弃边界改为 `outputId` 集合。

## 4. 第一版修正目标

第一版不追求复杂的端到端对话编排，只先解决明显体验问题。

### 4.1 推荐状态拆分

第一版只保留 3 个概念：

| 概念 | 含义 |
| --- | --- |
| 当前 assistant 输出 | 当前浏览器正在播或排队的 `outputId` |
| 用户插话候选 | 播放中本地检测到用户可能在说话 |
| 输出丢弃集合 | 当前应丢弃哪些旧 `outputId` 的 assistant 文本和音频 |

不要继续用单个布尔值表达“停播、等待用户、丢弃音频、丢弃文本”。文本和音频只按 `outputId` 同步。

### 4.2 推荐前端行为

语音打断：

1. 播放中检测到连续人声时，先降低或停止本地播放。
2. 继续上传麦克风音频，不关麦克风。
3. 等 ASR 文本回来后再决定是否发送上游 interrupt。
4. 如果 ASR 被判定为回声或语气词，恢复正常播放策略。

按钮打断：

1. 只有 assistant 正在输出时才可用，或无输出时点击不改变后端状态。
2. 点击后立即停止本地播放。
3. 界面提示“已停下，继续说”。
4. 后续用户输入进入新轮次。

静音：

1. 关闭麦克风采集。
2. 停止当前播报。
3. 不把它当成用户打断，不发送 `CLIENT_INTERRUPT`。

### 4.3 推荐后端行为

后端需要区分“旧输出残留”和“当前用户输入后的新回复”。

推荐做法：

- 每一轮 assistant 输出生成一个 `outputId`。
- assistant 文本和音频都带同一个 `outputId`。
- `interrupt_ack` 回传 `targetOutputId`，前端把它加入丢弃集合。
- 收到新的 ASR 或文字输入后，下一条 assistant 使用新的 `outputId`。
- 后端不长期 suppress assistant 文本。

## 5. 界面反馈建议

通话界面至少需要让用户知道当前处于哪种状态：

| 状态 | 建议文案 |
| --- | --- |
| AI 正在播报 | `正在说...` |
| 检测到用户插话 | `我停下了，你继续说。` |
| 等待用户继续输入 | `说吧，我在听。` |
| 静音 | `麦克风关，播报已停止。` |
| 打断失败 | `打断失败，请再说一遍或点结束通话。` |

系统事件可以保留给日志，但主界面不应只显示 `Interrupt: 用户已实时打断当前播报。` 这类工程化文本。

## 6. 验收标准

第一版可以按以下场景验收：

- AI 长句播报中，用户说“等一下”，本地声音在短时间内停下。
- AI 长句播报中，用户直接说一句新问题，不需要先说“停”。
- 用户只发出“嗯”“啊”等反馈音，不应频繁误打断。
- 用户点击打断按钮后，旧播报不再继续播放。
- 用户点击打断按钮后继续说话，assistant 能正常回应这次输入。
- 通话中点击静音，不发送上游打断，也不生成“用户打断”日志。
- 打断后旧 assistant 文本和旧音频不会重新出现在界面或扬声器里。

## 7. 暂不处理

以下内容不放进第一版：

- 多设备同时通话的跨端打断同步。
- 复杂的半双工/全双工模式切换配置。
- 用户个性化打断敏感度。
- 基于模型判断的高级 turn-taking。
- 长期统计模型或商业化质检指标。

第一版先把单个 Web H5 通话里的插话体验做顺，再考虑这些扩展。
