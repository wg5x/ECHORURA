# AI 播客化与文档播报玩法调研

调研日期：2026-06-10

## 结论

`ai-engine` 可以把“播客”作为 P2 内容产品能力来做，但第一阶段不要做成泛娱乐播客平台，也不要只是把文章用 TTS 读一遍。

更适合的方向是：

> 做一个“深度调研到播客化”的内容生成引擎，把一个题目、一个话题、一次搜索调研、或者一组企业文档，转成可听、可追问、可复用的音频内容。

核心机会不是替代真人播客，而是把原本需要阅读、整理、讲解的材料，变成几种可消费的音频形态：

- 一个话题的深度调研播客。
- 一组文档的双人解读播客。
- 一篇 PRD、报告、政策、教程的短播报。
- 一次会议、访谈、客服通话的会后音频摘要。
- 一个知识库的系列化课程音频。

第一版建议先做“文档 / 话题 -> 结构化稿件 -> 多角色播报音频 -> 引用和摘要回看”，而不是一开始做分发、订阅、创作者社区或广告变现。

## 为什么现在值得做

### 1. 用户已经接受“文档变播客”

Google NotebookLM 的 Audio Overview 已经把“上传资料 -> 两个 AI 主持人解读 -> 可下载收听”做成标志性形态。官方帮助文档把 Audio Overview 定义为 AI 主持人围绕用户上传来源生成的深度讨论，并支持 Deep Dive、Brief、Critique、Debate 等多种格式，还能指定语言、长度和关注点。

这个方向说明用户不只需要“朗读”，还需要：

- 把复杂材料讲清楚。
- 把多个来源串起来。
- 用对话降低理解门槛。
- 在通勤、做家务、运动时消费知识。

### 2. AI 音频工具正在从配音工具走向内容工作台

ElevenLabs 的 GenFM 支持从 PDF、文章、电子书、文档或导入文本生成个人播客，并由 AI co-host 播报。Wondercraft 则强调从 idea、doc、URL 或 raw ideas 生成脚本、配音、编辑、发布，并在 API 层提供多段 script + voice_id 的音频生成任务。

这说明“播客化”不再只是 TTS 能力，而是一个完整工作流：

1. 输入材料。
2. 提取观点。
3. 设计节目结构。
4. 生成主持稿。
5. 多音色演绎。
6. 后期包装。
7. 审核和发布。

`ai-engine` 如果只接 TTS，会停在能力层；如果把“调研、稿件、音色、引用、播放和追问”串起来，才是产品层。

### 3. 播客正在变成交互界面

Spotify 2026 年发布的新播客功能里，用户可以对正在听或看的播客提问，并实时获得回答；同时 Spotify 也在做 Personal Podcasts，让用户生成与兴趣和收听习惯相关的短私密音频。此前 Spotify 已经上线播客评论、Q&A、投票等互动能力。

这说明未来播客不是单向文件，而会变成：

- 用户可追问的内容界面。
- 根据个人兴趣生成的私密节目。
- 能把问题、投票、评论反馈给内容生产侧的交互产品。
- 能把一个长内容拆成多个可跳转、可继续探索的音频节点。

`ai-engine` 的实时语音链路正好可以承接“听完后继续问”“边听边打断”“把某段展开讲”的交互。

## 范围定义

本方案中的“播客化”指：

> 把题目、话题、链接、文档、知识库、会议材料或已有转写，通过调研、整理、改写、脚本生成和 TTS，转成单人或多人的音频节目，并保留来源引用、结构化摘要和可追问入口。

它和普通 TTS 的区别：

| 维度 | 普通 TTS 播报 | 播客化生成 |
| --- | --- | --- |
| 输入 | 一段确定文本 | 题目、文档、链接、知识库、会议转写 |
| 处理 | 按原文朗读 | 调研、归纳、重组、改写、分角色 |
| 音频形态 | 单人朗读 | 单人简报、双人对谈、辩论、讲课、问答 |
| 用户价值 | 不看屏幕也能听 | 帮用户理解、记忆、比较和复盘 |
| 可信度要求 | 原文准确 | 必须有来源、引用、人工审核和错误提示 |
| 后续交互 | 播放 / 暂停 | 追问、跳章节、生成续集、转文字笔记 |

## 豆包播客 API 能力判断

火山引擎豆包语音已经有独立的“播客API-websocket-v3协议”，不是只能用普通 TTS 拼双人音频。

官方文档信息：

| 项 | 内容 |
| --- | --- |
| 文档 | `播客API-websocket-v3协议` |
| 文档 ID | `1668014` |
| 首次发布 | 2025-07-07 |
| 最近更新 | 2026-05-12 |
| WebSocket 地址 | `wss://openspeech.bytedance.com/api/v3/sami/podcasttts` |
| Resource ID | `volc.service_type.10050` |
| 试用入口 | `https://console.volcengine.com/speech/service/10028` |

### 能直接支持的输入

| 模式 | 参数 | 能力 | 对 `ai-engine` 的意义 |
| --- | --- | --- | --- |
| 长文 / URL 总结播客 | `action = 0`，传 `input_text` 或 `input_info.input_url` | 根据长文本、网页链接、可下载 PDF/DOC/TXT 链接生成双人播客 | 适合快速验证“文档变播客”，但来源引用和事实审核要在产品层补 |
| 已有脚本合成播客 | `action = 3`，传 `nlp_texts` | 根据已拆好的轮次文本和发音人直接合成播客 | 最适合作为 MVP 主路径：先生成并审核脚本，再调用豆包合成音频 |
| 联网话题总结播客 | `action = 4`，传 `prompt_text` | 根据一个简单话题联网总结并生成播客 | 适合演示“一个题目变播客”，但 prompt 不具备复杂指令能力，不能替代自建调研链路 |

关键限制：

- `input_text` 上下文最长 32k，超过会报错。
- `action = 0` 下可以用 `input_info.input_text_max_length` 控制模型处理最大字符数，官方建议不超过 12000 更稳定，超过会截断，并在 `PodcastEnd` 的 `input_metrics` 里返回截断信息。
- `action = 3` 的 `nlp_texts.text` 单轮不超过 300 字符，总文本不超过 10000 字符。
- `speaker_info.speakers` 只能选 2 个发音人。
- `prompt_text` 不具备复杂指令能力，更像“话题词 / 简短问题”，例如“火山引擎”“怎么平衡工作和生活？”。

### 输出和事件

播客生成阶段客户端不需要持续发送上行 event，主要消费下行事件：

| Event | 含义 | 产品用途 |
| --- | --- | --- |
| `150 SessionStarted` | 会话任务开始 | 建立任务状态 |
| `360 PodcastRoundStart` | 新轮次开始，带轮次 idx 和 speaker | 生成章节、字幕和说话人时间轴 |
| `361 PodcastRoundResponse` | 当前轮次音频内容 | 流式播放或写入音频文件 |
| `362 PodcastRoundEnd` | 当前轮次结束，返回 `audio_duration`、`start_time`、`end_time` | 做章节边界和断点续传 |
| `363 PodcastEnd` | 播客结束，可带 `meta_info.audio_url` 和 `input_metrics` | 保存完整音频链接、截断信息和总结元数据 |
| `154 UsageResponse` | 返回文本输入 token 和音频输出 token | 做成本统计 |
| `152 SessionFinished` | 合成完成 | 任务完成状态 |
| `2 FinishConnection` / `52 ConnectionFinished` | 客户端关闭连接 / 关闭成功 | 正常释放连接 |

如果设置 `input_info.return_audio_url = true`，`PodcastEnd` 事件里会返回完整播客音频下载链接，但有效期只有 1 小时。产品侧仍需要把最终音频转存到自己的对象存储，并保存章节、脚本版本、音色、来源版本和审核状态。

### 音色和合规能力

豆包播客 API 支持：

- 专属播客音色。
- TTS 音色。
- ICL 复刻音色。
- 1.0 / 2.0 版本音色。
- `audio_config.format`：`mp3`、`ogg_opus`、`pcm`、`aac`。
- `audio_config.sample_rate`：16000、24000、48000。
- `audio_config.speech_rate`：-50 到 100。
- `input_info.strict_audit`：声明安全审核等级，默认 `false`，对 `action = 0` 的 `input_text` 和 `action = 4` 的 `prompt_text` 生效。
- `aigc_watermark`：结尾增加音频节奏标识。
- `aigc_metadata`：在 mp3/wav/ogg_opus 的音频 header 加入隐式水印元数据。

需要注意：复刻音色购买的 APPID 要和开通播客的 APPID 一致才能通过鉴权。也就是说，`voice_assets` 里必须记录“音色所属 APPID / 授权范围 / 是否可用于播客”，不能只存 speaker id。

### 对方案的修正

有了豆包播客 API 后，`ai-engine` 不需要第一版自己拼“多轮脚本 -> 多次 TTS -> 音频合成”。这个 API 的玩法更像“播客生成器 + 播客合成器”，不是单纯 TTS：

| 玩法 | 推荐阶段 | 做法 | 产品含义 |
| --- | --- | --- | --- |
| API 先生成播客文本 | POC / 演示 | `action = 0` 或 `action = 4`，设置 `input_info.only_nlp_text = true` | 快速把文档、URL 或话题变成双人播客轮次，用来验证内容形态 |
| 人工审核后再合成音频 | MVP 主路径 | 先保存并编辑 API 生成的轮次，再用 `action = 3` + `nlp_texts` 合成 | 把事实、合规、品牌口径和音色授权放在人审之后 |
| 直接生成完整音频 | 低风险内部场景 | `action = 0` / `action = 4` 直接生成音频，并按需设置 `return_audio_url = true` | 适合内部资料播报或 demo，不适合未审核公开发布 |
| 自建脚本后只借 API 合成 | 高控制场景 | `ai-engine` 自己生成结构化脚本，再用 `action = 3` | 适合客户交付、品牌栏目、需要严格来源引用的内容 |

更合理的落地路径是：

1. 第一阶段不要先做复杂脚本生成器，先接 `action = 0` / `action = 4` 的 `only_nlp_text`，拿到豆包生成的播客轮次。
2. 产品层把这些轮次保存成可审核脚本，允许人工修改、删除、重排和标记来源。
3. 审核后用 `action = 3` 把确认过的 `nlp_texts` 合成为双人音频。
4. 需要完整音频下载时设置 `input_info.return_audio_url = true`，拿到 1 小时有效链接后转存。
5. 用 `PodcastRoundStart` / `PodcastRoundEnd` 生成章节和说话人时间轴，用 `UsageResponse` 做成本统计。

这个 API 可以承担“把材料组织成播客轮次”的一部分工作，但不能替代 `ai-engine` 的产品层能力：来源管理、事实核查、人工审核、权限控制、音色授权、内容标识和后续可追问。

## 适合 `ai-engine` 的玩法

### 1. 深度调研播客

用户输入一个题目，例如：

- “AI 语音广告基础设施是否值得做？”
- “2026 年播客 AI 化的新产品形态有哪些？”
- “某个行业客户的智能客服机会在哪里？”

系统输出：

- 调研资料列表。
- 观点结构。
- 10-20 分钟双人播客稿。
- 3 分钟高管摘要版。
- 可回看的引用和结论卡片。

推荐定位：产品决策、行业研究、投资研究、竞品分析。

关键点：

- 不能只生成“听起来像研究”的内容，必须保留来源和不确定性。
- 播客脚本要先经过结构化大纲，再生成口语稿。
- 默认输出中文，必要时可生成中英双语版。

### 2. 文档解读播客

用户上传一组文档，例如：

- PRD、架构设计、需求评审材料。
- 政策、合同、操作手册。
- 课程讲义、论文、行业报告。
- 企业内部知识库页面。

系统输出：

- “这份文档讲什么”的 2 分钟 Brief。
- “重点拆解”的 8-15 分钟 Deep Dive。
- “风险和反对意见”的 Critique。
- “正反方讨论”的 Debate。

推荐定位：企业内部学习、项目交接、评审前预习、客户材料解读。

第一版可以只做 Markdown、PDF 转写文本、网页正文和会议转写文本；P2 再接知识库目录、权限同步和批量更新。

### 3. 文档播报 / 通勤简报

用户选择部分文档或一个文件夹，系统每天生成短播报：

- 今天新增了什么。
- 哪些事项需要我处理。
- 哪些结论和上次不一样。
- 哪些风险变高了。

推荐定位：管理者晨间播报、研发项目日报、销售客户跟进、法务政策更新。

它和播客的区别是更短、更稳定、更任务导向。适合作为 `ai-engine` 的低成本入口，因为它主要依赖摘要和 TTS，不一定需要复杂多角色脚本。

### 4. 会议 / 通话复盘播客

基于已有 ASR 或会议转写，生成会后音频：

- “刚才会议的 3 个结论”。
- “双方分歧在哪里”。
- “谁负责什么事项”。
- “下次会议前要补哪些材料”。

推荐定位：销售复盘、客服质检、项目会议、面试复盘。

这个玩法和现有实时语音链路很近：`ai-engine` 已经能拿到 ASR、会话事件、用量和错误，后续只需要把会话日志进入“会后脚本生成 + TTS”链路。

### 5. 课程化系列播客

把一个知识库或一组文档拆成系列音频：

- 第 1 集：背景和核心概念。
- 第 2 集：常见误区。
- 第 3 集：案例讲解。
- 第 4 集：测验和复盘。

推荐定位：企业培训、产品培训、销售 enablement、开发者教育。

这类玩法不建议放在 MVP，因为需要章节规划、学习进度、测验和内容运营。但它是后续商业价值比较高的方向。

### 6. 角色化品牌播客

把固定音色、固定主持人人设和固定栏目模板绑定起来：

- 产品经理主持人：用适合评审的方式拆解 PRD。
- 技术主持人：解释架构、风险和依赖。
- 客服主持人：把客户问题转成服务复盘。
- 品牌主播：用品牌音色播报新品、活动、教程。

推荐定位：品牌内容、客户成功、IP 化知识服务。

它依赖音色资产和声音授权，必须复用 `voice_assets` 的授权、过期、撤回和使用范围控制，不允许把克隆音色当作普通参数。

### 7. 可追问播客

用户听到某段内容后可以问：

- “刚才提到的第三点展开讲。”
- “这个结论来自哪份文档？”
- “换成给老板听的版本。”
- “把这段生成一条待办。”

推荐定位：`ai-engine` 的差异化能力。

普通音频生成工具通常到“生成音频文件”为止；`ai-engine` 可以把播客播放器和实时语音 Agent 打通，让播客成为知识交互入口。

## 推荐产品形态

### MVP：文档播客生成台

目标：证明“文档 / 话题可以稳定变成高质量可听内容”。

用户流程：

1. 输入题目，或上传 / 粘贴 1-5 份材料。
2. 系统抽取来源、观点、术语和冲突点。
3. 用户选择节目格式：Brief、Deep Dive、Critique、Debate、播报。
4. 系统先调用播客 API 生成轮次文本，必要时再用自建模型补充大纲和脚本。
5. 用户在页面上人工检查脚本。
6. 用户选择音色并生成音频。
7. 页面展示音频、章节、摘要、来源引用和下载入口。

MVP 只做：

- 资料输入。
- API 轮次文本生成。
- 脚本审核和轻编辑。
- 审核后调用豆包播客 API 生成双人音频。
- 章节和来源回看。
- 人工确认后生成音频。

MVP 不做：

- 开放播客分发平台。
- 用户订阅和推荐流。
- 创作者社区。
- 广告变现。
- 自动发布到 Spotify / Apple Podcasts。
- 未审核脚本直接自动生成并外发。

### P1：播报自动化

目标：把一次性生成变成可重复的内部播报。

能力：

- 文件夹或知识库监听。
- 每日 / 每周摘要。
- 只播新增和变化。
- 生成待办和风险提醒。
- 支持“短版 / 长版 / 老板版 / 执行版”。

适合从项目文档、客户材料、会议纪要里生成固定节奏的音频简报。

### P2：可追问播客

目标：让用户听完后能继续问。

能力：

- 音频章节和来源段落对齐。
- 播放到某段时保留上下文。
- 用户语音追问。
- 追问结果可生成补充音频或文字卡片。
- 支持“继续展开这一段”“跳到反对意见”“只讲和我相关的部分”。

这里需要和实时语音主链路结合，形成 `Podcast Player + Realtime Voice` 的产品形态。

### P3：系列化和品牌化

目标：形成可售卖的企业内容产品。

能力：

- 栏目模板。
- 固定主持人和品牌音色。
- 多语言版本。
- 团队审核流。
- 权限控制。
- 播放数据和学习数据。
- 客户成功 / 培训 / 销售 enablement 场景包。

## 推荐架构

```mermaid
flowchart LR
  Input["题目 / 文档 / 链接 / 转写"] --> Ingest["资料解析与权限校验"]
  Ingest --> Research["调研与要点抽取"]
  Research --> Outline["节目大纲"]
  Outline --> Script["脚本生成"]
  Script --> Review["人工审核"]
  Review --> Voice["音色与角色选择"]
  Voice --> TTS["TTS 生成音频"]
  TTS --> Package["章节 / 摘要 / 引用 / 音频包"]
  Package --> Player["播放器"]
  Player --> Realtime["实时语音追问"]
  Realtime --> Research
```

模块拆分：

- `Content Ingest`：解析 Markdown、PDF 文本、网页正文、会议转写，保留来源和权限。
- `Research Planner`：根据题目或文档生成调研问题、观点结构和待核查点。
- `Evidence Store`：保存来源片段、引用、可信度和使用范围。
- `Podcast Script Generator`：把结构化大纲转成可播报稿，不直接调用 TTS。
- `Review Workflow`：人工确认事实、口径、语气和不该说的内容。
- `Voice Casting`：选择音色、角色、语速、情绪和语言。
- `Audio Renderer`：调用 TTS，生成音频、章节时间轴和字幕。
- `Interactive Player`：支持播放、章节跳转、来源回看和语音追问。

## 生成质量要求

### 内容质量

- 每个关键结论都要能回到来源或标记为推断。
- 不把来源没有说的内容包装成事实。
- 不把多个来源的矛盾强行合并成确定结论。
- 输出要有“已知事实 / 分析判断 / 仍需确认”三类口径。
- 播客稿要口语化，但不能牺牲准确性。

### 听感质量

- 单段不要过长，适合 20-60 秒一个信息块。
- 双人对话要有真实分工：一个负责解释，一个负责追问或挑战。
- 不要频繁寒暄、夸张口头禅和无意义捧哏。
- 技术文档要先讲背景，再讲结构，再讲风险。
- 汇报类内容要先给结论，再展开依据。

### 产品质量

- 生成音频前必须能预览脚本。
- 生成后必须能看到章节、摘要和来源。
- 每个音频都要记录使用的模型、音色、来源版本和审核人。
- 对外发布前必须有内容审核状态。
- 用户必须知道这是 AI 生成或 AI 合成音频。

## 合规和风险边界

必须避免：

- 未经授权使用真人、名人、员工、客户的克隆音色。
- 把内部文档生成可外发音频。
- 把不确定结论播成确定事实。
- 生成医疗、法律、金融等高风险建议并省略免责声明。
- 生成冒充真人主持人的播客。
- 未标识 AI 生成合成音频。

需要产品约束：

- 音色资产必须走授权状态。
- 来源文档必须有权限范围。
- 外发前必须有审核记录。
- 高风险领域默认只允许生成“资料解读”，不允许生成“具体建议”。
- 音频元数据中保留 AI 生成标识和来源版本。

## 商业化方向

更适合先做 B 端 / 团队内产品，而不是 C 端公开播客社区。

| 方向 | 付费方 | 价值 |
| --- | --- | --- |
| 企业知识库播报 | 企业团队 | 让员工更快吸收文档、政策、培训材料 |
| 项目 / 管理层简报 | 管理者 | 把文档变化、会议结论转成可听摘要 |
| 销售和客户成功播客 | 销售团队 | 把客户资料、竞品和方案变成拜访前音频 |
| 培训课程音频化 | HR / 培训团队 | 降低课程生产成本，支持多语言 |
| 品牌内容工作台 | 市场团队 | 快速生成品牌播客、产品播报和活动音频 |

不建议早期做：

- 播客广告平台。
- 开放内容广场。
- 主播打赏。
- 音频社交。
- 泛娱乐 AI 主播平台。

这些方向对内容供给、审核、分发和社区治理要求很高，会偏离 `ai-engine` 的语音能力底座定位。

## 与现有语音路线的关系

播客化能力依赖现有路线里的三条基础能力：

1. 独立 TTS：生成长音频、短播报、角色化解读。
2. 音色资产：管理官方音色、品牌音色、授权克隆音色。
3. 实时语音：支持听后追问、打断、继续展开。

推荐顺序：

1. P0 先把实时语音主链路稳定下来。
2. P1 接独立 TTS 和音色资产。
3. P2 做文档播客生成台。
4. P2+ 做会后复盘音频和每日播报。
5. P3 再做可追问播客、系列课程和品牌栏目。

## 示例

### 示例 1：从题目生成深度调研播客

输入：

> 2026 年播客 AI 化有哪些新玩法？哪些适合 ai-engine 做？

输出：

- 标题：AI 播客正在从内容文件变成知识界面
- 格式：Deep Dive，双人对谈，中文，12 分钟
- 章节：
  - 00:00 结论：不是朗读文章，而是资料理解界面
  - 01:30 NotebookLM 类“文档变讨论”
  - 04:00 ElevenLabs / Wondercraft 类“内容生产工作台”
  - 06:30 Spotify 类“播客可追问和个性化”
  - 09:00 `ai-engine` 该先做什么
  - 11:00 风险：事实、版权、音色、审核
- 交付物：
  - 音频文件
  - 脚本文本
  - 参考来源
  - 3 条产品建议

### 示例 2：从 PRD 生成评审前播报

输入：

> 上传 `prd-voice-agent.md`、`architecture.md`、`risk-review.md`

输出：

- 2 分钟 Brief：适合评审前快速听。
- 10 分钟 Critique：指出需求边界、技术依赖和风险。
- 文字卡片：
  - 本次 PRD 的目标。
  - 还没确认的问题。
  - 可能过度设计的地方。
  - 必须在评审会上拍板的事项。

### 示例 3：从会议转写生成复盘播客

输入：

> 一次客户需求会的转写文本

输出：

- 客户真正要解决的问题。
- 我方承诺过的事项。
- 客户没有明说但可能在意的风险。
- 下次沟通前需要准备的材料。
- 5 分钟销售复盘音频。

## 近期落地建议

1. 新建 `Podcast Project` 概念，不直接把它塞进实时语音会话。
2. 第一版只支持手动上传 / 粘贴材料，避免先做复杂知识库同步。
3. 脚本必须先生成并可编辑，审核后再调用豆包播客 API。
4. 先支持两种格式：`brief` 和 `deep_dive`。
5. 先支持 2 个固定主持角色，不做开放角色市场。
6. 音频生成记录来源版本、音色、模型和审核状态。
7. 播放器先做章节、字幕、来源回看；追问放到下一阶段。
