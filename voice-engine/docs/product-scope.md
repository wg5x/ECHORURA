# Voice Engine 产品边界

## 1. 一句话定位

Voice Engine 是一套通用语音行动引擎，面向视觉障碍、低操作门槛和移动场景用户，提供“语音输入 -> 语义路由 -> 业务执行 -> 流式反馈”的可复用能力。

ECHORURA 音乐创作是第一条业务场景。用户可以通过说话、哼唱和少量确认动作完成：


- 生成 30-60 秒音乐草稿。
- 用语音修改歌词、风格、人声、节奏和情绪。
- 选择或复用模板。
- 发布到 ECHORURA 内测作品页。
- 记录作品版本、模板来源和二创链路。

## 2. MVP 做什么

通用 Voice Engine 第一版做：

1. S2S 实时语音对话入口。
2. 音色、模式、system role、联网、唱歌等会话参数抽离。
3. S2S 事件、payload、延迟和错误记录。
4. 会话、turn 和长任务进度管理。
5. 语义路由和参数抽取。
6. H5 与 Native Shell 的 JS Bridge。
7. 白名单动作、风险判断和二次确认。
8. 业务事件流协议。

音乐创作场景第一版只做一个尖锐闭环：

1. 用户用语音描述创作意图。
2. 系统理解意图并生成创作任务。
3. 音乐生成服务返回草稿。
4. 用户继续用语音修改。
5. 用户选择是否保存为作品或模板。
6. 系统生成平台内作品页记录。
7. 后台记录作品、模板、版本和来源关系。

## 3. MVP 不做什么

通用 Voice Engine 暂不做：

- VAD / 打断，直到技术方案和可测试方案明确。
- 任意第三方 App UI 自动化。
- 取代所有业务系统的通用超级助手。
- 无白名单的系统级动作执行。
- 云手机自动操作第三方 App 作为主路径。
- 对第三方 App 内部 UI 的任意自动化控制。

音乐创作场景暂不做：

- 专业 DAW 时间线、多轨编辑、插件生态。
- 钱包前台化、DAO、二级市场、复杂版权交易。
- 全网音乐上架和正式版权发行。
- 多级链上自动分成结算。

## 4. 核心语音意图

| 意图 | 示例 | 下游 |
|---|---|---|
| `chat` | “这个功能怎么用？” | Chat Agent |
| `navigate` | “打开我的作品页。” | H5 Router / Native Bridge |
| `native_action` | “分享这首歌。” | Native Capability Adapter |
| `server_action` | “帮我保存这个草稿。” | Server Tool / Business Service |
| `record_idea` | “先记下这个灵感。” | Draft Service |
| `clarify` | “你想生成什么风格？” | Clarify Agent |
| `reject` | 高风险或不支持请求 | Safety Response |

## 5. 用户体验要求

- 用户必须能在少看屏幕或不看屏幕的情况下完成关键流程。
- 每一步关键状态都要有语音反馈，例如“已生成草稿”“正在修改副歌”“已保存”。
- 长任务必须先反馈进度，不能让用户沉默等待。
- 发布、删除、外部分享、付费等动作必须二次确认。
- 所有按钮、结果卡片、错误状态都要兼容屏幕阅读器。
- H5 页面必须支持键盘导航、ARIA 标签、焦点管理和可播报状态。

## 6. 成功标准

通用 Voice Engine 验证指标：

- 语音转写和路由是否稳定。
- H5 与 Native Shell 的 Bridge 是否可靠。
- 用户是否理解语音反馈和确认流程。
- 高风险动作是否能正确拦截并确认。

音乐场景 MVP 验证指标：

- 用户是否能用语音完成一首 30-60 秒作品。
- 从语音输入到第一版草稿是否能控制在 2-5 分钟。
- 用户是否会进行第二次语音修改。
- 用户是否愿意保存或发布作品。
- 用户是否愿意使用模板。
- 模板是否产生复用和二创链路。
