## 2. 已形成的核心共识

### 2.1 产品方向

ECHORURA 不再以“大而全 Web3 音乐经济系统”作为第一阶段主线，而是收敛为：

> 面向视觉障碍与低操作门槛创作者的语音 AI 音乐创作工具。

核心体验：

- 用户说出创作意图。
- 系统生成 30-60 秒音乐草稿。
- 用户用语音修改歌词、风格、人声、节奏和情绪。
- 用户选择或复用模板。
- 系统发布到平台内作品页。
- 后台记录作品、模板、版本和二创链路。

### 2.2 Web3 角色

Web3 不进入第一层用户体验，不先讲钱包、Gas、DAO、资产和二级市场。

Web3 第一阶段只承担：

- 作品 hash 记录。
- 模板来源记录。
- 版本关系记录。
- 二创链路记录。
- 默认分成规则记录。

### 2.3 技术主路径

`voice-engine` 已经明确采用 H5-first + Android Native Shell 架构：

- H5 承载主要业务页面，Android WebView 和 PC Web 复用同一套页面。
- Android Native Shell 负责麦克风、TTS/ASR、权限、Intent、分享、文件选择和无障碍增强。
- 服务端负责 Voice Gateway、Semantic Router、Scenario Skill Router、业务任务编排和事件流。
- 音乐创作是首个 `music_creation` 场景，不进入 Voice Engine 核心。
- 云手机 / mobile-use 不作为 MVP 主路径，只作为远期远程自动化兜底。

这个方向与 AI App Lab 参考中的判断一致：语音入口 + Agent 路由应优先考虑 RTC 对话式 AI / ASR / TTS / VAD / CustomLLM，而不是直接把云手机作为主要执行层。

### 2.4 豆包实时语音接入共识

第一阶段不直接把 S2S 当作业务大脑，而是参考火山官方 `rtc-aigc-demo`：

- 语音体验层复用 RTC、ASR、TTS、VAD、打断和音频 3A。
- `CustomLLM` 回调接入 ECHORURA 的 Voice Gateway。
- Voice Gateway 内部完成意图识别、参数抽取、二次确认和场景路由。
- S2S-Omni / S2S-SC 放到第二阶段评估，用于更自然的陪伴式语音反馈；涉及发布、分享、上架、分成、支付等业务动作时，仍必须转成结构化事件后执行。

## 3. 当前 `voice-engine` 文档理解

### 3.1 产品边界

`voice-engine/docs/product-scope.md` 已经定义了 MVP 做什么和不做什么。

MVP 做：

- 语音描述创作意图。
- 生成音乐草稿。
- 语音修改。
- 保存为作品或模板。
- 生成平台内作品页。
- 记录作品、模板、版本和来源关系。

MVP 不做：

- 专业 DAW 时间线。
- 钱包前台化。
- DAO、二级市场、复杂版权交易。
- 全网音乐上架。
- 多级链上自动分成。
- 云手机自动操作第三方 App 作为主路径。

### 3.2 技术架构

`voice-engine/docs/technical-architecture.md` 的主链路是：

```text
H5 工作台 / Android WebView / PC Web
  -> Android Native Bridge
  -> 实时语音层
  -> Voice Gateway
  -> Semantic Router
  -> Chat / Native Action / Server Action / Scenario Skill
  -> 流式语音反馈
```

核心模块：

- H5 工作台：承载业务页面，复用到 Android WebView 和 PC Web。
- Android Native Shell：提供语音、权限、Intent、分享、文件选择和 TalkBack 增强。
- 实时语音层：ASR、TTS、VAD、打断、降噪，可选 RTC。
- Voice Gateway：会话、turn、取消和流式事件管理。
- Semantic Router：意图识别、参数抽取、风险判断。
- Scenario Skill Router：分发到音乐、内容创作、教育、客服等场景。
- Music Creation Skill：音乐生成、模板、发布和 lineage 记录。
- Template Service：模板选择与继承关系。
- Publish Service：保存和发布作品页。
- Lineage Ledger：记录作品、模板、版本、二创链路。
- Web3 Adapter：音乐场景的可选后台能力，用于 hash / 来源 / 分成规则记录。

### 3.3 协议草案

`voice-engine/specs/voice-command-event-protocol.md` 已经定义了核心消息类型：

- `route_decision`
- `native_action`
- `scenario_command`
- `task_progress`
- `task_result`
- `business_event_recorded`
- `confirmation_required`

后续实现时，应该优先把这个协议转成可校验的 schema，例如 JSON Schema 或 TypeScript / Kotlin / Python 类型定义。

### 3.4 ADR

`voice-engine/adr/0001-android-first-voice-engine.md` 的主路径已修正为 H5-first + Native Shell。

关键原因：

- 业务页面应优先 H5 化，以便 Android WebView 和 PC Web 复用。
- 视觉障碍体验依赖端侧无障碍能力。
- 云手机会引入登录态、隐私、成本和合规复杂度。
- Android Intent / Deep Link / Native Action 更适合由 Native Shell 通过 JS Bridge 暴露给 H5。

## 4. 下一阶段落地计划

### 4.1 第 0 阶段：文档收口

目标：让产品、技术、协议、验证计划对齐。

待办：

- 更新根目录 `README.md`，加入 `voice-engine` 工程索引。
- 补齐 `voice-engine/docs/assets/diagrams/voice-engine-architecture.mmd` 到文档链接。
- 把 `voice-engine/specs/voice-command-event-protocol.md` 中的 JSON 示例沉淀为 schema。
- 明确 MVP 的 3-5 个语音命令优先级。
- 明确 `rtc-aigc-demo` 到 Voice Gateway 的最小改造点：场景 JSON、CustomLLM 回调、Gateway 入参、Gateway 返回可播报文本和事件。

建议优先语音命令：

1. `create_song`
2. `revise_song`
3. `choose_template`
4. `publish_work`
5. `record_idea`

### 4.2 第 1 阶段：Mock Demo

目标：不接真实音乐生成模型，先跑通语音到作品页的闭环。

最小实现：

- H5 + Android WebView Shell 或 Web 原型采集语音。
- ASR 转文本。
- Gateway 接收文本。
- Router 输出 `route_decision`。
- Mock Music Provider 返回固定音频 URL、歌词、标题、封面。
- 前端播报“已生成草稿”。
- 用户语音触发保存或发布。
- Lineage Ledger 写入本地或数据库。

验收：

- 用户能用语音完成一次 create -> revise -> publish。
- 每一步都有可播报反馈。
- 发布动作有二次确认。
- 作品记录中包含 `work_id`、`version_id`、`template_id`、`source`。

### 4.3 第 2 阶段：真实音乐生成接入

目标：接入真实音乐生成 Provider，但保持 Provider 可替换。

待办：

- 定义 `MusicGenerationProvider` 接口。
- 支持创建任务、查询进度、获取结果。
- 音频、歌词、封面、元数据进入 Asset Store。
- 任务进度映射成 `task_progress`。
- 生成结果映射成 `task_result`。

注意：

- 不要把架构绑定到某一家 AI 音乐服务。
- 长任务必须流式反馈，不能沉默等待。
- 失败时要提供可播报错误和下一步建议。

### 4.4 第 3 阶段：无障碍可用性测试

目标：验证视觉障碍用户是否能独立完成主流程。

测试任务：

- 说出创作意图。
- 听到系统确认。
- 生成并试听草稿。
- 语音修改一次。
- 保存或发布作品。
- 使用一个模板。

观察指标：

- 不看屏幕完成率。
- 平均完成时间。
- 用户请求帮助次数。
- TalkBack 读屏是否清晰。
- 用户是否愿意再次创作。

### 4.5 第 4 阶段：模板与 Lineage

目标：验证模板复用和二创链路。

待办：

- 定义模板数据结构。
- 定义作品版本数据结构。
- 记录 `template_id -> work_id`。
- 记录 `work_id/version_id -> new_version_id`。
- 生成可哈希事件，为后续 Web3 Adapter 做准备。

### 4.6 第 5 阶段：Web3 Adapter 后台化

目标：只做轻量记录，不把钱包暴露给普通用户。

待办：

- 生成 lineage event hash。
- 定义可上链事件格式。
- 选择测试链或先本地模拟。
- 不做二级市场，不做外部收益承诺。

## 6. 近期最重要的决策

### 6.1 先 H5 Demo 还是 Android WebView Shell

文档已决策 H5-first + Native Shell。如果投资人演示时间紧，可以先做 Web Demo 验证页面和流程，再接 Android WebView Shell 的 Native 能力。

建议：

- 投资人 Demo：Web 原型可以接受。
- 用户可用性测试：必须回到 Android WebView / TalkBack。

### 6.2 实时语音方案

可选：

- 端侧 ASR/TTS + HTTP/SSE：实现简单，延迟略高。
- RTC 对话式 AI + CustomLLM：体验更好，适合长期方向。

建议：

- MVP 技术验证先用端侧或云 ASR/TTS + WebSocket/SSE。
- 真正面向视觉障碍用户时评估 RTC + VAD + 打断。

### 6.3 音乐生成 Provider

短期不要锁死供应商。先抽象接口，用 mock 跑通，再接真实 Provider。

### 6.4 Web3 链选择

暂不急。先定义 lineage event 和 hash。链上写入可以在后续验证模板复用后再做。

## 7. 投资人/团队沟通话术

可以这样解释现在的阶段：

> 我们已经把 ECHORURA 从“大而全的 Web3 音乐平台”收敛为一个更可落地的语音行动引擎。Voice Engine 本身保持通用，负责语音、路由、H5/Native Bridge、确认和事件流；音乐创作是第一个场景，用来验证视觉障碍和低操作门槛用户是否能通过语音完成创作、修改、模板复用和发布。

技术上可以这样讲：

> 主路径采用 H5-first + Android Native Shell。业务页面用 H5 实现，PC 和 Android WebView 复用；Android 原生层只提供语音、权限、Intent、分享、文件选择和无障碍增强。Voice Engine 核心是通用语音行动引擎，音乐创作是首个场景。
